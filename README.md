# Realtime Multilingual Chat Foundation

This phase implements a room-based multilingual text chat foundation using:

- React + Vite + Tailwind CSS frontend
- Python FastAPI backend
- WebSockets for room-scoped message broadcasting
- `langdetect` for message language detection
- Local LibreTranslate in Docker for self-hosted translation
- A modular translation service ready for later AI integration

It intentionally does not include authentication, a database, WebRTC, voice/video, or an admin panel.

## Run LibreTranslate Locally

Install Docker Desktop first if Docker is not already available:

```powershell
docker --version
docker compose version
```

Start LibreTranslate from the project root:

```powershell
docker compose up -d libretranslate
```

The first startup may take a while because LibreTranslate downloads local language models into the Docker volume.

Verify the container is running:

```powershell
docker compose ps
```

Verify the API:

```powershell
curl http://localhost:5000/languages
```

Optional direct translation check:

```powershell
curl -X POST http://localhost:5000/translate -H "Content-Type: application/json" -d "{\"q\":\"Hello world\",\"source\":\"en\",\"target\":\"hi\",\"format\":\"text\"}"
```

FastAPI reads LibreTranslate from:

```text
LIBRETRANSLATE_URL=http://127.0.0.1:5000
```

That default works for local development. Set `LIBRETRANSLATE_URL` only if you expose LibreTranslate on a different host or port.

## Verify room-based chat

1. Start the backend.
2. Start the frontend.
3. Open `http://localhost:5173` in two browser tabs.
4. Join the same room from both tabs with different usernames and languages.
5. Send a message from one tab.
6. The sender should see the original message.
7. Other users should see a language-specific LibreTranslate result.
8. Open a third tab and join a different room.
9. Messages from the first room should not appear in the third tab.

## WebSocket translation flow

1. The user enters a username, room ID, and preferred language in the React app.
2. The React app opens `ws://localhost:8000/ws/{room_id}/{user_lang}`.
3. On socket open, the frontend sends a join payload with username and room ID.
4. FastAPI stores the websocket, username, room ID, and preferred language in memory.
5. Chat messages are sent as JSON: `{ "text": "...", "sender_name": "...", "room_id": "..." }`.
6. The backend detects the original language with `langdetect`.
7. For each receiver in the room, the backend calls local LibreTranslate with the detected source language and receiver target language.
8. The sender gets the original text as `translated`.
9. Other users get a translated text generated by the local Docker container.
10. The frontend displays `translated` prominently and `original` below it.
11. When a browser disconnects, the backend removes it from the room and broadcasts a leave notification.

## FastAPI to LibreTranslate Flow

```text
Frontend
  -> FastAPI WebSocket
  -> backend/app/translation/service.py
  -> LibreTranslate Docker container on http://localhost:5000
  -> FastAPI WebSocket
  -> Receiver browser
```

Translation failures are handled inside the translation service. If LibreTranslate is unreachable or returns an invalid response, the backend logs a warning and sends the original text so the chat room keeps working.

## Hardened Translation Architecture

The translation module is organized around small, unit-testable functions in `backend/app/translation/service.py`:

- `detect_language_profile(text)` detects the source language and flags mixed Hindi/English messages.
- `should_translate(source_lang, target_lang)` prevents unnecessary same-language translation.
- `build_cache_key(text, source_lang, target_lang)` creates deterministic cache keys.
- `extract_translated_text(response_payload)` isolates LibreTranslate response parsing.
- `translate_text(...)` returns structured translation metadata with status, errors, cache hits, and mixed-language flags.

Translation status is included in each message payload:

```json
{
  "message_kind": "text",
  "original": "Hello world",
  "translated": "Hola mundo",
  "detected_language": "en",
  "target_language": "es",
  "translation_status": "success",
  "translation_error": null,
  "cache_hit": false,
  "mixed_language": false
}
```

Current translation statuses:

- `success`: LibreTranslate returned translated text.
- `skipped_same_language`: sender and receiver selected the same language.
- `fallback`: LibreTranslate returned a known bad artifact, such as a language label.
- `failed`: LibreTranslate was unavailable or returned an invalid response.

This keeps the message shape ready for future voice/video signaling by preserving `type` and adding `message_kind`.

## Future translation integration point

The best place to integrate a custom or local AI translation model is `backend/app/translation/service.py`. Keep the same async `translate(text, target_lang, source_lang)` interface and replace the LibreTranslate implementation with a local multilingual model, a fine-tuned translation pipeline, or a service trained on a custom dataset.
